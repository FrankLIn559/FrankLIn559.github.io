1.前言

一般而言，web服务分为静态和动态两种。服务器收到由浏览器发起的HTTP请求后，遵循着web容器 -> 连接数据库 -> 生成静态页面 -> 返回数据这样的流程来响应用户请求的，就是动态页面。另一方面，服务器在收到请求后，无需访问数据库而直接从文件系统中得到事先准备好的html文件返回给用户的，就是静态页面。

毫无疑问，动态页面的功能更加强大，也更加灵活，目前互联网上可以访问到的绝大多数网站都是动态的。但是动态博客系统的情况却不太一样：

动态博客系统虽然有着诸如响应速度快、能够随时随地在线编辑、操作简单等这样那样的优点，但同时也存在着一些缺点：

1.环境不易搭建：动态博客系统的搭建相对而言比较复杂，而且linux系统不同的发行版之间难免有些差异，即使一步一步跟着网上的教程走也不一定能一下子搞定 -> 难搞哦。

2.成本比较高：一般一台个人服务器。要想安稳运行并被外界访问的话还得购买一个可备案域名，否则就只能通过ip地址来访问网站了。

3.备份麻烦。服务器是租来的而不是自己的，忘记续费了的话，服务提供商保留一段时间后就会收回这台主机 -> 意味着数据被清空。而且，万一机房哪天遇到个天灾人祸什么的，我们这些苦逼的用户就犹如晴天霹雳、被劈的外焦里嫩喽，当然这种事情发生的概率还是比较小的，而且大点的云服务厂商都会配套成熟的数据库备份系统。
除了上面说的，博客系统系统的选择还要考虑到以下几点：

1.在内容上：重视文字、轻视数据，不过分依赖于数据库。
2.在功能上：需求稳定，无需频繁修改业务逻辑。
3.在速度上：主要由作者自己的需求而定，可高可低。
对于程序员而言，选择诸如Hexo、Jekyll、Hugo这样的静态博客系统也不失为一种好办法，因为我们会Markdown、会Git。

3. 安装 Hexo

什么是Hexo？
Hexo 是一个静态的、简洁且高效的博客框架，其功能就是帮助用户自动生成静态页面，叫它博客页面生成器也未尝不可。其最大的优点就是操作简单、托管方便，用户只需使用Markdown来进行创作并将写好的文章放入指定目录下，在几秒内Hexo就能帮助你生成一系列简洁、美观的静态页面。

各文件/文件夹用途：
_config.yml：整个网站的配置信息。

package.json：记录该项目使用到的所有依赖包，比如用来部署网站的deployer-git和用来渲染markdown文件的 renderer-markdown-it，都是依赖项。

node_modules：依赖包文件的实际存放目录（package.json只是个配置文件，这里才是真正放文件的地方）。

scaffolds：新建文章时自动使用的模板，是指在新建的markdown文件中默认填充的内容。

source: 资源文件夹，其中_post目录用来存放markdown文件 - 自己写的文章。

themes：主题文件夹，Hexo会根据主题来生成静态页面。

public：公共文件夹，存放生成的站点文件，是最终用户在浏览器上看到的内容。

_config.yml常用配置项：
网站相关：

参数 描述
title 网站标题
description 网站描述，主要用于告诉SEO(Search Engine Optimization，搜索引擎优化)一些基本信息，增加被搜索到的几率
language 网站使用的语言
timezone 网站时区， 默认使用你电脑的时区。比如说：America/New_York，Japan，和 UTC
url 当网站存放在子目录里面时使用，在根目录的话不用配置，会自动匹配
root 重新指定根目录

目录相关：

参数 描述 默认值

public_dir 公共文件夹，用于存放生成的站点文件 public
tag_dir 标签文件夹 tags
archive_dir 归档文件夹 archives
category_dir 分类文件夹 categories

文章相关：

参数 描述 默认值

5.更换主题

是不是觉得自己的网站很丑，没关系加个主题就行了，Hexo官网里有个主题商店，收录了许多主题，往往质量都比较高，客官请进去随意挑选！

当然也有一些开发者，没有选择把自己的主题发布到主题商店，而是仅仅托管在自己的GitHub之上，其中也不乏一些优秀的主题。

选择主题
如何选择主题呢？我觉得是这样的：

1.选择文档详尽的。没有好的文档的主题不要轻易使用。
2.选择用户基数大的。如果你选择的是一款小众主题，假设作者放弃维护了，那么你的体验就会变得极其糟糕。而比较出名的主题，作者一般不会随便放弃维护的。
安装主题
把主题文件整个下载下来（手工下载、git clone都行），放在themes文件夹下（里面已经有个默认的landespace了，不用管它），然后再_config.yml中切换使用的主题就行了，一般这些主题的文档里都会写得清清楚楚的，如果存在和我说的有冲突的地方，听他的！

6.备份

我们的远程仓库master分支上存放的是生成的站点文件，可我们的配置文件、资源文件等还没加入版本控制系统呢，为了避免重装系统、格式化等情况出现导致博客环境丢失，我们最好在远程仓库上再新建一个hexo分支，把所有的环境配置文件push到这个分支以作备份。觉得有点吃力的同学，也可以选择再新建一个仓库来存储它们，实在不行U盘备份都行啊，反正千万不要让环境裸奔。
